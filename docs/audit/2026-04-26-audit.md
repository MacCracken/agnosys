## Methodology

### Local code-pattern review

Pattern scans across `src/` and `lib/` for:

- `sys_system` / shell exec via `/bin/sh -c` (command injection vector)
- `syscall(59)` / `sys_execve` raw exec (no argv validation)
- `var buf[N] >= 4096` inside fn scope (cyrius static-data hazard from CHANGELOG 1.0.1 / vidya field notes)
- `var buf[N] >= 65536` (binary bloat + static-data hazard at CI threshold)
- Writes to `/etc/`, `/bin/`, `/sbin/` outside of read-only paths
- Path-traversal via `..` in caller-supplied paths
- File-open syscalls (`sys_open`, `read_file`) without validation

### External research

WebSearch for current CVE landscape, focused on the kernel surfaces agnosys exposes. See "External CVE Landscape" below for citations. The audit is dated **2026-04-26**; CVEs from 2025 H2 and 2026 Q1 are the primary concern.

### Tooling baselines

- `cyrius capacity --check`: all tables ≤ 7% utilization (well under 85% gate)
- `cyrius vet src/main.cyr`: include-graph clean
- `cyrius lint src/*.cyr`: 0 warnings
- `cyrius test`: 222 / 222 passed
- `scripts/audit.sh`: all 10 gates pass
- Fuzz: 3 harnesses (`audit_nlmsg`, `certpin_pin`, `pam_config`), 500 iters / 10 s each, no crashes

## External CVE Landscape (2025 H2 – 2026 Q1)

The Linux kernel team became a CVE Numbering Authority in 2024; disclosure rates have accelerated dramatically (3,529 kernel CVEs in 2024; CISA added 7 to the KEV catalog in 2025). The list below is the subset directly relevant to agnosys's binding surface.

### Netlink (audit.cyr, netns.cyr)

- **CVE-2026-31495** — ctnetlink netlink policy validation flaw. Allows manipulation of conntrack tables, potential privilege escalation. *Relevance:* agnosys's `audit_*` and `netns_*` modules issue netlink syscalls; defensive validation of reply messages matters.
- **CVE-2026-31407** — netfilter conntrack netlink validation, out-of-bounds memory access. *Relevance:* netns nft surface.
- **CVE-2026-31664** — xfrm netlink padding leak (kernel heap memory disclosed via netlink multicast). *Relevance:* informational; agnosys does not subscribe to xfrm multicast.
- **CVE-2026-31416** — `nfnetlink_log` netlink message-loss bug due to size miscalculation. *Relevance:* informational.
- **CVE-2026-31531** — IPv4 nexthop netlink sizing bug (large ECMP groups). *Relevance:* informational.
- **CVE-2026-23231** — UAF in `nft_chain` via netlink rule manipulation. Requires capability to manipulate nftables. *Relevance:* netns nft callers run as root (required for namespace creation); a compromise upstream of agnosys could weaponize the chain UAF — agnosys is not the vulnerability sink but is on the data path.

### PAM (pam.cyr)

- **CVE-2025-6018** & **CVE-2025-6019** — PAM + libblockdev/udisks privilege escalation chain. Local user → root across major distros. Disclosed by Qualys, June 2025. *Relevance:* agnosys's pam.cyr is read-only inspection (PAM config parsing, passwd/who); does not invoke PAM stack. Informational only.
- **CVE-2025-6020** & **CVE-2025-8941** — `pam_namespace` path-traversal via symlink race. Affects pam_namespace ≤ 1.7.0. Local-root via TOCTOU. *Relevance:* agnosys does not call pam_namespace, but the **symlink-race / predictable-tmpfile** pattern is class-of-bug for any module writing to `/tmp` (see F-3).
- **CVE-2026-3888** — Ubuntu systemd-tmpfiles + snap-confine privilege escalation. *Relevance:* agnosys is not on the path; informational.

### LUKS / dm-verity / TPM (luks.cyr, dmverity.cyr, tpm.cyr)

- **LUKS2 confidential-VM vulns (Trail of Bits, Oct 2025)** — malleable LUKS2 metadata headers allow attacker to coerce TEE guests into encrypting with `cipher_null-ecb` (which returns plaintext unchanged). cryptsetup v2.8.1 has partial mitigation. *Relevance:* **direct.** agnosys's `luks_cipher_str` does not allowlist cipher names. See F-2.
- **TPM2 SPI bus sniffing (gbhackers / oddlama Jan 2025)** — `TPM2_NV_Read` keys sent in plaintext over SPI without session protection. Physical-attack class. *Relevance:* agnosys's tpm.cyr issues PCR reads via the device; does not establish parameter-encryption sessions. Out of agnosys's scope (TPM2 session management is a higher-level concern; sigil owns the policy).
- **systemd-cryptsetup TPM2 issues (issue #39049)** — TPM2 unlock fails without system change; informational.

### FUSE (fuse.cyr)

No newly-disclosed CVEs in scope window matched; FUSE remains a historically high-volume area but agnosys's binding is read-only `/proc/mounts` parsing + mount/unmount syscalls (no userspace FUSE filesystem implementation).

### IMA / Secure Boot (ima.cyr, secureboot.cyr)

No newly-disclosed CVEs matched in the search window for these subsystems specifically. IMA appraisal and Secure Boot rely on trusted-boot chains where the failure modes are typically configuration / policy issues, not memory-safety bugs.

### Landlock / seccomp (security.cyr)

No 2025-2026 CVEs found matching agnosys's surface. Both subsystems are unprivileged sandboxing primitives; the typical "escape" patterns are kernel bugs unrelated to the userspace API agnosys exposes.

### General Linux kernel exploitation in 2025

- **CVE-2025-21756** ("Attack of the Vsock") — VM-to-host root via vsock subsystem. *Relevance:* informational; agnosys does not bind vsock.
- 7 kernel CVEs added to CISA KEV in 2025 — confirms the threat model that "kernel privilege escalation" is the operative risk for agnosys's consumers.

## Per-Module Findings

### F-1 — HIGH: `journald_query` shell injection via `/bin/sh -c` — **FIXED in 1.0.1**

**Module:** `src/journald.cyr` (`journald_query`)
**Class:** Command injection (CWE-78)
**Status:** **FIXED.** `journald_query` now calls `lib/process.cyr::exec_capture(args, buf, buflen)` with an argv vec built by `_journald_build_argv(filter)`. Every filter value lands in its own `argv[i]` and is read verbatim by journalctl — no shell, no metacharacter expansion. Public helper `journald_build_args/1` was removed (was only useful as a shell-injection input format) and the snapshot updated. Fuzz coverage: `fuzz/journald_filter.fcyr` exercises shell-metacharacter inputs through every filter setter and asserts argv shape — passes 500-iter / 10-s gate.

```cyr
fn journald_query(filter) {
    var cmd_str = journald_build_args(filter);
    # ...
    var res = run_capture("/bin/sh", "-c", str_data(cmd_str), buf, 131072);
```

`journald_build_args` constructs a single space-separated argument string from caller-controlled `filter` setters: `journald_filter_set_unit(f, unit)`, `_set_grep(f, grep)`, `_set_since(f, since)`, etc. The string is then passed to `/bin/sh -c`, which performs **full shell expansion** including `;`, `|`, `&`, `` ` ``, `$()`, redirection, etc.

**Attack path:**
```
journald_filter_set_unit(f, "foo; rm -rf /tmp/* #")
journald_query(f)
→ /bin/sh -c "journalctl --output=json --no-pager --unit=foo; rm -rf /tmp/* # ..."
```

A consumer that passes any string from external input (HTTP API, config file, environment variable) to a filter setter without sanitization is one chained call away from arbitrary command execution running with the consumer's privileges (which for argonaut may be root).

**Why it exists:** the comment at `journald.cyr:118` documents the choice — `lib/process.cyr`'s `_exec3` only takes 2 extra args, so the module bundles all journalctl flags into one string and shells out. This was a workaround for the missing `argv[]` API.

**Recommendation:**

1. Add `run_capture_vec(args, buf, buflen)` to `lib/process.cyr` mirroring `exec_vec` (vec-of-strings → argv → execve, no shell). One-time lib change.
2. Refactor `journald_query` to push individual journalctl flags into a vec and call `run_capture_vec`; delete `journald_build_args` and the `/bin/sh -c` path.
3. Until (1)/(2) ship: document the footgun in `docs/SECURITY-NOTES.md` and recommend consumers sanitize against shell metacharacters `; | & \` $ ( ) < > \ ' "` before passing to filter setters.

**Tracking:** Phase 8 patch slot. Not blocking 1.0.x.

### F-2 — MEDIUM: LUKS cipher allowlist absent — **FIXED in 1.0.1**

**Module:** `src/luks.cyr` (`luks_cipher_str`, `luks_config_validate`)
**Class:** Insufficient input validation (CWE-20)
**Status:** **FIXED.** New public fn `luks_validate_cipher(algo, mode)` allowlists `aes`/`serpent`/`twofish`/`camellia` paired with `xts-plain64` / `cbc-plain64` / `xts-essiv:sha256` / `cbc-essiv:sha256`. Case-insensitive `null` substring rejection catches `cipher_null`, `Null`, `NULL`, `cipher_null-ecb` (Trail of Bits 2025 attack vector), case variants, and any name containing `null` anywhere. Wired into `luks_config_validate`, which is called by both `luks_format` and `luks_open`, so existing callers automatically get the protection. Fuzz coverage: `fuzz/luks_cipher.fcyr` exercises positive cases, every shape of null/identity attack, off-list algos/modes, case variants, empty/null pointer edge cases — passes 500-iter / 10-s gate.

```cyr
fn luks_cipher_str(config) {
    var algo = luks_config_cipher_algo(config);
    var mode = luks_config_cipher_mode(config);
    # ... build "<algo>-<mode>" with no validation
}
```

Per the Trail of Bits Oct 2025 disclosure, `cipher_null-ecb` is a real attack vector against LUKS2: cryptsetup accepts it, and the kernel cipher_null implementation returns plaintext unchanged. A malicious `algo="cipher_null"` / `mode="ecb"` from a config slot produces a working `cipher_null-ecb` argument to `cryptsetup luksFormat --cipher`.

**Status:** cryptsetup v2.8.1 ships partial upstream mitigation. agnosys should add its own allowlist as defense-in-depth, since consumers may run on systems with older cryptsetup.

**Recommendation:** add `luks_validate_cipher(algo, mode)` that:
- Allowlists known-safe (algo, mode) pairs: `(aes, xts-plain64)`, `(aes, cbc-plain64)`, `(serpent, xts-plain64)`, `(twofish, xts-plain64)`
- Explicitly rejects: `cipher_null`, `null`, any algo containing `null`, identity transforms
- Returns `Err(invalid_argument)` on rejection
- Called from every public LUKS fn that consumes a cipher (`luks_format`, `luks_open` via the `cipher` field)

Add a fuzz harness `fuzz/luks_cipher.fcyr` exercising the allowlist.

**Tracking:** 1.0.x patch slot recommended (security-relevant; defense-in-depth against an active disclosure).

### F-3 — LOW: predictable temp-file path in LUKS keyfile — **FIXED in 1.0.1**

**Module:** `src/luks.cyr` (`luks_keyfile_path`, `luks_write_keyfile`)
**Class:** Insecure temporary file (CWE-377), symlink-race (CWE-59)
**Status:** **FIXED.** `luks_keyfile_path` now appends 8 random bytes (16 hex chars) from `getrandom(2)` to the per-PID prefix → `/tmp/.agnos-luks-<pid>-<16hex>`. Returns `Result(path)` so callers handle a getrandom failure (callers updated). `luks_write_keyfile` opens with `O_WRONLY | O_CREAT | O_TRUNC | O_EXCL | O_NOFOLLOW` (mode 0600) — `O_EXCL` refuses to open an existing path (closes the race window), `O_NOFOLLOW` refuses to traverse a symlink at the final component (closes the CVE-2025-6020 / pam_namespace symlink-race class).

```cyr
fn luks_keyfile_path() {
    var pid = sys_getpid();
    # ... returns "/tmp/.agnos-luks-<pid>"
}
```

PID is enumerable. A local attacker on the same system can pre-create `/tmp/.agnos-luks-<predicted_pid>` as a symlink to e.g. `/etc/shadow`. When agnosys writes key material to that path via the keyfile flow, a non-O_EXCL open follows the symlink and either disclosure or destruction occurs. CVE-2025-6020 is a recent same-class bug in `pam_namespace`.

**Recommendation:**
- Open with `O_CREAT | O_EXCL | O_NOFOLLOW` (refuse to follow symlinks; refuse to overwrite existing files)
- After open, `fstat` and verify `S_IFREG` and ownership matches `geteuid()`
- Add randomness to the suffix beyond PID (e.g., 8 bytes from `getrandom(2)`); current PID-only naming is broken when two LUKS operations race

**Tracking:** 1.0.x patch slot.

### F-4 — LOW: secureboot fallback path bogus — **FIXED in 1.0.1**

**Module:** `src/secureboot.cyr` (`secureboot_sign_module`)
**Class:** Logic bug (not security; flagged because the `..` pattern looked suspicious)
**Status:** **FIXED.** Hardcoded `/usr/lib/modules-load.d/../linux/scripts/sign-file` (which resolved to a path that exists on no supported distro) replaced with runtime detection: `agnosys_uname` to get the running kernel release, then `sys_access` probe for `/lib/modules/<release>/build/scripts/sign-file` first, falling back to `/usr/src/linux/scripts/sign-file`. If neither exists the error message names both paths so the operator can install the right package.

```cyr
vec_push(args2, "/usr/lib/modules-load.d/../linux/scripts/sign-file");
```

`/usr/lib/modules-load.d/../linux/scripts/sign-file` resolves to `/usr/lib/linux/scripts/sign-file`, which does not exist on Arch / Debian / Fedora / Ubuntu / openSUSE. Kernel `sign-file` lives at `/usr/src/linux/scripts/sign-file` or `/lib/modules/$(uname -r)/build/scripts/sign-file` depending on distro.

**Not a security issue** (the path is a hardcoded constant, not user input), but the `..` is misleading and the fallback is broken on every distro tested. If `kmodsign` (the primary path) succeeds, this code is unreachable; if `kmodsign` fails, the fallback also fails and we return `err_io("kernel module signing failed")` — so the user never gets module signing, but never gets a security bug either.

**Recommendation:** either fix the path (probably `/lib/modules/$(uname -r)/build/scripts/sign-file`, requires runtime detection of kernel version) or drop the fallback entirely and improve the kmodsign error message. Track as Phase 8 cleanup.

### F-5 — LOW: 4 KB static-data buffers in dmverity / ima output paths — **VISIBILITY FIX LANDED**

**Modules:**
- `src/dmverity.cyr:1699,1803` (`dmverity_format`, `dmverity_status`) — `var outbuf[4096]`
- `src/ima.cyr:3129` (`ima_get_status`) — `var rbuf[4096]`

**Class:** Cyrius static-data hazard (vidya field-note `var_buf_in_library_functions`).

Cyrius emits `var buf[N]` in fn scope as **static data, not stack**. Two consecutive calls share the same backing memory. If a fn returns a `Str` borrowing into the buffer, the second call's contents are visible through the first call's return value.

Code review:
- `dmverity_format` reads `outbuf` → parses content → returns `Ok(0)` (no buffer reference). **SAFE.**
- `dmverity_status` reads `outbuf` → constructs a heap-allocated `dmverity_status` struct (alloc'd separately, contents copied via memcpy). **SAFE.**
- `ima_get_status` reads `rbuf` → similar pattern (parsed into struct, no Str borrow). **SAFE.**

**No vulnerability today.** Flagged for ongoing vigilance: any future refactor that replaces `memcpy`/parse with a returned `str_new(rbuf, n)` pointing into the static buffer would silently introduce the bug. The 1.0.1 CI security gate flags `var buf[N] >= 65536`; **lower the threshold to `>= 4096` on the next CI iteration** so this class of buffer is at least visible in CI logs.

**Recommendation:** lower the security-scan threshold in `.github/workflows/ci.yml` from 65536 → 4096 with the existing match-but-don't-fail behavior, OR migrate these three sites to heap-allocated buffers (`alloc(4096)` + free or arena). Phase 8 cleanup.

### F-6 — Defense-in-Depth: audit netlink reply type/flags not verified — **FIXED in 1.0.1**

**Module:** `src/audit.cyr` (`audit_get_status`)
**Status:** **FIXED.** `audit_get_status` reply parsing now validates `nlmsg_len` against bytes received, requires `nlmsg_seq == 1` (the request seq), surfaces `NLMSG_ERROR` (type 2) as a structured `err_syscall_failed` with the negative-errno extracted from the payload, and rejects any other unexpected `nlmsg_type` instead of silently parsing it as audit_status. Constants `NLMSG_NOOP` / `NLMSG_ERROR` / `NLMSG_DONE` added to the audit module's enum block. Fuzz coverage: `fuzz/audit_reply.fcyr` synthesizes 8+ classes of malformed reply (truncated, oversized claim, wrong seq, NLMSG_ERROR, bogus types, exact boundaries) and asserts the validation rejects each — passes 500-iter / 10-s gate (random-type stress confirms only AUDIT_GET with seq=1 and ≥24-byte payload accepts).

`audit_get_status` correctly checks `nbytes < NLMSG_HDRLEN + 24` before parsing. It does **not** verify:
- `nlmsg_type` of the reply matches the request (`AUDIT_GET`); a malformed reply with an unexpected type and the same length is parsed anyway
- `nlmsg_flags` (specifically `NLM_F_MULTI` / `NLMSG_DONE` flow control)
- Sequence number matches the request

Per CVE-2026-31495 (ctnetlink) and the broader pattern of netlink validation flaws, defensive parsers should validate the full nlmsghdr, not just length. Today the kernel doesn't generate misbehaving replies for AUDIT_GET — but kernel CVEs have historically been the source of exactly this class of malformed message.

**Recommendation:** in every audit netlink reply parser:
1. Verify `nlmsg_type` field matches the expected response type (or `NLMSG_ERROR`)
2. Verify `nlmsg_seq` matches the request seq
3. On `NLMSG_ERROR`, parse the error code and return a structured `err_*` instead of treating as success

Phase 8 audit-module hardening slot. Add `fuzz/audit_reply.fcyr` covering malformed reply shapes (truncated, wrong type, wrong seq).

### F-7 — Defense-in-Depth: netns nftables ruleset pass-through

**Module:** `src/netns.cyr` (nftables ruleset application)

`netns_apply_nftables_ruleset` accepts a string ruleset from the caller and feeds it to `nft -f -` via `exec_vec`. The nft binary parses the ruleset; agnosys does not. Per CVE-2026-31407 (netfilter conntrack netlink validation) and CVE-2026-23231 (UAF in nft_chain), the kernel-side surface is exposed to whatever the caller writes.

**Status:** agnosys is correct here — argv-based exec, no shell injection, ruleset content is intentionally pass-through (consumers like nein own the policy). The kernel nft surface is the consumer's risk to manage.

**Recommendation:** document in `docs/SECURITY-NOTES.md` that nftables ruleset content is pass-through; consumers must trust their ruleset source. No agnosys-side change.

### F-8 — Confirmed safe: `var buf[120]` in syscall.cyr

**Modules:** `src/syscall.cyr:152,160,168` (`agnosys_uptime`, `agnosys_total_memory`, `agnosys_free_memory`)

Each fn declares `var buf[120]` (the sysinfo struct), calls `query_sysinfo(&buf)`, and returns `Ok(sysinfo_uptime(&buf))` — an **integer payload**, not a borrow. The buffer is fn-local and not referenced after return. **No vidya-class hazard.** No action.

## What Is Explicitly OK

The audit confirms the following surfaces are safe as of 1.0.1:

- **No `sys_system` calls.** All process invocations use `exec_vec` (argv-based) or `_exec3` (positional argv). No shell expansion except F-1.
- **No raw `syscall(59)` execve.** All exec goes through `lib/process.cyr`.
- **No writes to `/etc/`, `/bin/`, `/sbin/`.** Reads only (passwd, shadow, pam.d, group).
- **PAM module string `..` is rejected** at `pam.cyr:615` (defensive, opposite-sign of the bug).
- **No buffers ≥ 64 KB.** Largest is `pam.cyr` 16 KB and `journald.cyr` 128 KB — both heap-allocated via `alloc()`, not `var buf[N]`.
- **Capacity gate green:** all compiler tables ≤ 7% utilization. Plenty of headroom for Phase 8.
- **All 222 integration tests pass.** Fuzz harnesses (audit_nlmsg, certpin_pin, pam_config) clean at 500 iters / 10 s.
